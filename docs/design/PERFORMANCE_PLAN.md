# PERFORMANCE PLAN

This is the next performance plan for the current renderer and asset pipeline.
The game now treats local assets as the normal path: optimized files live under
`public/assets/` and runtime code loads them through `/assets/...` URLs. The
work below makes that path faster, easier to measure, and safer as the asset set
grows.

Companions:

- `OPTIMIZATION.md` and `OPTIMIZATION_SPEC.md` cover the earlier sim and render
  hot-path pass.
- `GRAPHICS_SPEC.md` covers the visual target and asset policy.
- `ASSETS.md` records provenance for every shipped asset.
- `PROGRESS.md` records measured results once work lands.

## 0. Current Baseline

The renderer already has several good foundations.

- Assets are local same-origin files under `public/assets/`. Source code refers
  to them with plain runtime strings like `/assets/creeps/wolf.glb`.
- `src/engine/asset-loaders.ts` caches model, texture, and HDR promises by URL.
- Models are optional. If a GLB, JPG, or HDR is absent or fails to load, the
  procedural fallback stays live.
- Terrain props use `InstancedMesh` once authored foliage and rock GLBs load.
- Creep GLBs are loaded once per URL, cloned per unit, and mounted over the
  procedural rig.
- Audio is procedural WebAudio, so there is no audio download cost today.
- Quality presets already gate post-processing, shadows, environment maps,
  pixel ratio, VFX caps, and weather density.

The next bottlenecks are likely to be asset size, decode time, GPU upload,
shader compilation, draw calls, unique materials/textures, animation mixer cost,
and cache policy. Local files help network reliability, but they do not remove
those costs.

## 1. Principles

- **Measure first.** Every milestone needs a before and after number. Use the
  browser for GPU/frame-time evidence and the headless harness for stable CPU
  regression checks.
- **Keep assets local.** Ship optimized assets from `public/assets/`, with raw
  source packs left in `tmp/asset_src/` and documented in `ASSETS.md`.
- **Keep fallbacks real.** The game should still boot and play with
  `public/assets/` empty. Low tier should be able to skip heavy GLBs.
- **Pay for visible work.** Load, animate, and shade what the player can see or
  will see soon. Defer the rest.
- **Preserve the core boundary.** Asset, render, and loading work stays out of
  `src/core/`.

## 2. Workstreams

### 2.1 Asset Delivery

Make the local asset path explicit and cache-friendly.

- Add an asset manifest generated by `scripts/assets/build_assets.mjs`, written
  beside the optimized files or as `public/assets/manifest.json`.
- Record URL, byte size, type, source spec, and optional preload group:
  `region`, `terrain`, `creep`, `town`, `hero`, `env`, `ui`.
- Keep source code loading by URL. Do not import binary assets into `src`; the
  no-asset-import guard stays useful.
- Version assets by filename or manifest hash before enabling long-lived cache
  headers in production. Example: `wolf.ab12cd.glb` or a manifest version key.
- Serve GLB, JPG, WebP, HDR, JS, and CSS with compression where the host supports
  it. Brotli helps text and some JSON. Meshopt GLBs are already compact, but
  headers still matter.
- Add an asset budget check that fails if committed `public/assets/` grows past
  agreed limits without an explicit update.

Acceptance:

- A generated manifest lists every shipped asset and its size.
- `ASSETS.md` and the manifest agree on the files that ship.
- Production cache headers can be long-lived without stale asset risk.

### 2.2 Asset Optimization

Keep decode and upload cost under control as the asset library grows.

- Continue using glTF Transform with `resample`, `prune`, `dedup`, texture
  resizing, and meshopt compression.
- Add per-category size targets:
  - Creep GLB: small enough to load several archetypes during region entry.
  - Town/foliage prop GLB: small enough for fast instancing.
  - Hero GLB: budgeted separately because animation and materials are heavier.
  - Terrain texture set: keep at 1K unless a profile proves 2K earns the cost.
- Convert embedded model textures to WebP where quality holds.
- Evaluate KTX2/Basis for terrain and model textures once image decode or GPU
  upload shows up in profiles. Keep the transcoder local if adopted.
- Normalize materials during the build step. Fewer unique materials usually
  means fewer shader variants and fewer draw calls.
- Strip unused animation clips. Keep only clips the animator can drive.
- Add a report mode to `build_assets.mjs`: total bytes, largest files, texture
  dimensions, material count, mesh count, animation count.

Acceptance:

- Every asset build prints a size and complexity report.
- New asset packs have clear byte budgets before they are committed.
- Largest files are easy to see in review.

### 2.3 Loading And Prewarming

Move unavoidable loading and compilation work away from the first playable
frame.

- Define preload groups from the manifest. Example: entering a forest region
  preloads the active terrain PBR set, the HDRI, forest foliage, rocks, town
  buildings, and the creep archetypes likely to spawn nearby.
- Keep preloading shallow. Load the current region and near-future assets, not
  the whole game.
- Add optional progress reporting to the loading screen: bytes discovered, bytes
  loaded, current group label.
- Use stale-load guards. If the player leaves a region while a GLB is still
  loading, the promise may resolve, but it should not mount into an obsolete
  scene.
- Prewarm shader programs for the loaded region after assets mount. The current
  loading screen already hides first-frame compilation; extend that to authored
  GLB materials.
- Prefetch likely next-region assets after the game is idle and stable for a few
  seconds. Cancel or ignore the work if memory pressure or a quality tier says
  to stay light.

Acceptance:

- Region entry has a visible loading state with optional progress.
- First interactive frame after region entry has no major shader or PMREM hitch.
- Stale asset loads cannot mutate a disposed scene.

### 2.4 Runtime Asset Cache

The current promise caches are a good start. They need lifecycle rules once more
regions and heroes ship.

- Keep a strong cache for assets used by the current scene.
- Keep a small warm cache for recently used or likely next assets.
- Release cold assets on region change when memory grows past a threshold.
- Dispose GPU resources when evicting: textures, geometries, materials, and
  PMREM outputs.
- Track reference counts or scene ownership for cloned models so shared source
  assets are not disposed while a clone is still alive.
- Add debug counters: loaded model count, texture count, approximate GPU texture
  bytes, cache hits, cache misses, evictions.

Acceptance:

- Re-entering a region hits the warm cache.
- Leaving several regions in a row does not grow GPU memory without bound.
- A debug panel can explain what is loaded and why.

### 2.5 Renderer And GPU Cost

Hold the frame budget as the scene becomes more authored.

- Measure draw calls, triangles, materials, textures, shader programs, and frame
  time in a browser debug overlay.
- Keep foliage, rocks, and repeated town dressing instanced.
- Bucket static props by material where possible.
- Add distance LOD for authored GLBs:
  - Full near the camera.
  - Reduced animation rate or simplified scene farther away.
  - Procedural fallback or hidden beyond far distance.
- Avoid mounting high-detail creep GLBs for units that start far outside the
  camera focus.
- Use shadow tiers aggressively. Many props can receive shadows without casting
  them, and low tier should skip expensive shadow work.
- Keep post-processing tiered. Bloom and grade are high value; AO stays high or
  ultra only.
- Add adaptive pixel ratio downshift when sustained frame time misses budget.
  Prefer slow, stable adjustment over constant oscillation.

Acceptance:

- The default tier holds the target scene at about 60 fps on the target machine.
- Low tier avoids GLB-heavy and post-heavy paths where needed.
- The debug overlay shows whether bottlenecks are draw calls, triangles, shader
  variants, or pixel cost.

### 2.6 Animation And Unit Views

Animated authored models can become expensive even when loading is solved.

- Keep one loaded source scene per GLB URL and clone per mounted unit.
- Share animation clips from the loaded asset. Do not duplicate clip data.
- Update animation mixers only for visible or near-visible units.
- Keep the current LOD rule: full units animate every frame, reduced units
  animate less often, culled units skip cosmetic animation.
- For far creeps, consider staying procedural until the player gets close.
- Audit skinned mesh counts and bone counts per hero/creep asset during the
  asset build report.
- Avoid per-frame material mutation on shared materials. Clone materials only
  when a unit needs unique opacity, tint, or effect state.

Acceptance:

- A 30-unit scene with authored creeps does not exceed the animation CPU budget.
- Far units do not run full skeletal animation.
- Shared source assets stay shared after repeated spawns/despawns.

### 2.7 Scene Lifecycle

Prevent leaks and stale objects as scenes are rebuilt.

- Add a scene disposal checklist: terrain, water, weather, post passes,
  environment maps, transient VFX, unit views, loaded authored clones, and
  debug-only objects.
- Add a stress test path: enter region A, travel to B, return to A, repeat.
  Watch object counts and GPU memory.
- Guard async `.then(...)` callbacks with scene tokens so old promises cannot
  attach to a new scene.
- Keep fallbacks hidden rather than disposed only while the authored model is
  mounted. Dispose them when the whole unit view is destroyed.

Acceptance:

- Region cycling does not steadily increase renderer object counts.
- Destroyed scenes receive no late mounts from async asset loads.

### 2.8 Browser Measurement

The headless harness protects CPU regressions. Browser evidence is still needed
for real graphics performance.

- Add a dev-only graphics HUD with:
  - frame ms average and p95,
  - draw calls,
  - triangle count,
  - texture count,
  - shader/program count if available,
  - loaded asset bytes from the manifest,
  - quality tier and active DPR.
- Add a fixed browser smoke route:
  1. new game,
  2. enter a busy region,
  3. spawn or walk into a 30-unit fight,
  4. hold for 60 seconds,
  5. record p95 frame time and cache stats.
- Keep manual tools in the loop for deeper issues: Chrome Performance panel,
  Memory panel, and Spector.js for draw-call inspection.

Acceptance:

- The browser smoke has a repeatable script and recorded numbers.
- `PROGRESS.md` records baseline and after numbers for each performance pass.

## 3. Milestones

### M1: Manifest And Budgets

- Generate `public/assets/manifest.json`.
- Add asset size report output.
- Add a size-budget check for `public/assets`.
- Record current totals in `PROGRESS.md`.

Done when the repo can answer: "what assets ship, how large are they, and which
group loads them?"

### M2: Loading Plan

- Define preload groups by region and asset type.
- Wire loading progress into the existing loading screen.
- Add stale-load scene tokens.
- Prewarm authored GLB materials before the loading screen fades.

Done when region entry is smooth and late asset loads are harmless.

### M3: Cache Lifecycle

- Add cache ownership rules and eviction.
- Add dispose paths for cold assets.
- Add debug counters for cache and GPU resource state.

Done when repeated region travel does not leak resources.

### M4: Renderer Budget Pass

- Add the graphics debug overlay.
- Tune shadows, post-processing, DPR, and GLB mounting by tier.
- Add adaptive DPR downshift behind a setting.
- Profile a 30-unit authored-creep fight in browser.

Done when the default tier holds target frame time and low tier has a clear
escape path.

### M5: Authored Model Scale-Up

- Apply asset build reports to each creep and hero GLB.
- Add LOD and animation throttling for authored models.
- Flip on only assets that meet size, material, and animation budgets.

Done when adding more local assets improves fidelity without moving the frame
budget backward.

## 4. Measurement Targets

Record the actual numbers when work starts. Initial targets:

- Default tier: about 60 fps in a 30-unit, 200-projectile scene on the target
  machine.
- p95 frame time: under 20 ms during normal combat, with short loading hitches
  hidden behind the loading screen.
- Region entry: no frozen first playable frame after the loading screen fades.
- Steady-state render hot path: no per-frame geometry or material allocation.
- Asset cache: no unbounded GPU memory growth after repeated region travel.
- Asset size: every new pack has a declared budget before commit.

## 5. Decisions To Log

Log these in `DECISIONS.md` as they land:

- Asset manifest format and whether filenames or manifest versions carry cache
  busting.
- Production cache-header policy.
- Per-category asset size budgets.
- KTX2/Basis decision, if adopted.
- Cache eviction policy and disposal ownership model.
- Preload groups and when background prefetch runs.
- Adaptive DPR thresholds and cooldown.
- LOD distances for authored creeps and heroes.
